Yes — ChatGPT Enterprise/Edu now supports **custom MCP-based apps/connectors**, so in principle your Bullhorn ATS use case is feasible even without a native Bullhorn integration. OpenAI’s current documentation says you can build your own custom app using MCP, connect it to your internal tools and data, and — in beta for Business/Enterprise/Edu — support both read/fetch and write/modify actions with admin controls and user confirmations for writes.[1][2]

## What exists today

OpenAI has renamed “connectors” to “apps,” but the key capability remains: custom apps can be built with Model Context Protocol so ChatGPT can retrieve information and call approved tools against external systems. Their help docs explicitly say custom apps can connect ChatGPT to your own tools and internal data, and their developer-mode article says full MCP support includes write/modify actions for Business, Enterprise, and Edu on ChatGPT web.[2][1]

For Enterprise/Edu, apps are disabled by default and must be enabled and published by admins; after that, admins can control access with RBAC and can configure which actions are allowed. OpenAI also says write actions require explicit user confirmation before execution, which is important for any Bullhorn field updates or record changes.[1][2]

## How Bullhorn fits

Bullhorn appears technically suitable for this pattern because its REST API supports OAuth-based authentication and then a REST session login flow returning a `BhRestToken` and a `restUrl` for subsequent API calls. Bullhorn’s docs also note production integrations should expect session expiry and handle refresh/re-auth flows, which aligns with OpenAI’s guidance that OAuth/OIDC setups for custom apps should be configured to issue refresh tokens when possible.[3][2]

That means the likely architecture is: ChatGPT Enterprise ↔ your custom remote MCP server ↔ Bullhorn API. The MCP server would expose controlled tools like “get candidate,” “search job orders,” “read placement,” and later “update candidate custom field” or “append note,” while your server handles Bullhorn auth, validation, logging, and safe translation between natural-language requests and Bullhorn API operations.[2][3][1]

## Likely implementation path

This does **not** look like a simple “paste API credentials into a wizard and Bullhorn just works” setup unless a third party has already built a Bullhorn MCP app for you. OpenAI’s setup flow for custom apps requires you to provide an MCP server endpoint plus metadata and authentication configuration, which strongly suggests you need either an internal engineering build or an integration partner rather than only a no-code form.[2]

A realistic phased rollout would be:
- Phase 1: Read-only tools, such as retrieving records, summarizing candidate/job history, and answering operational questions from Bullhorn data.[3][2]
- Phase 2: Narrow write actions, such as updating a small set of approved fields, creating notes, or changing statuses with confirmation prompts.[2]
- Phase 3: Broader workflow actions, with admin action controls, parameter constraints, and RBAC to limit who can do what. OpenAI says Enterprise/Edu admins can configure allowed actions and, for non-sync apps, apply parameter constraints to action arguments.[1]

## Practical opinion

From a research standpoint, your idea is **viable**, but the answer is “yes, with custom integration work,” not “yes, out of the box for Bullhorn.” OpenAI’s current platform gives the missing piece you were asking about — a custom MCP connector/app capability with both read and eventual write support — but Bullhorn access would still need to be packaged behind a remote MCP server and governed carefully by your Enterprise admins.[3][1][2]

The strongest production approach would be to start with read-only Bullhorn tools, prove value, then add a very small number of write actions with confirmation, auditability, and strict parameter validation. That lines up with OpenAI’s current controls and reduces the operational risk of letting ChatGPT modify ATS records too broadly too soon.[1][2]
